第三章 泛型编程风格

这一章主要介绍泛型编程。作者并没有在一开始就给出泛型的定义，而是循序渐进地，当我们当前的工具不足以实现更好，更灵活的编程时，他才引入泛型编程的介绍，更能知其所以然。这里的灵活是是指，当我们需要改变功能或者增添新的需求时，只需要修改几行代码，而不是整个地创建一个新函数去达成对应的目标。在引入介绍泛型算法之后，作者举例说明如何根据不断变化的需求去设计泛型算法。之后，便是常见数据Map和Set的用法介绍。最后是Iterator Inserter和iostream Iterator的用法说明。

书本内容摘抄：

STL概述

Standard Template Library (STL)主要由两种组件构成：一是容器(container)，包括vector、list、set、map等类；另一种组件是用以操作这些容器的所谓泛型算法(generic algorithm)，包括find()、sort()、replace()、merge()等。

vector、list这两种容器是顺序性容器(sequential container)。顺序性容器会依次维护第一个、第二个……，直到最后一个元素。我们主要在顺序性容器上进行迭代(iterate)操作。map和set这两种容器属于关联容器(associative container)。关联容器可以让我们快速查找容器中的元素值。

Function Objects

C++标准程序库预先定义了很多function objects。所谓function objects，是某种class的实体对象，这类classes对function call运算符进行了重载操作，如此一来，可使function object被当成一般函数来使用。

function object实现出我们原本可能以独立函数加以定义的事物。但又何必如此呢？主要是为了效率。我们可以令call运算符成为inline，因而消除“通过函数指针来调用函数”时需付出的额外代价。

标准程序库事先定义了一组function objects，分为算术运算(arithmetic)、关系(relational)、逻辑运箅(logical)三大类。以下列表中的type在实际运用时会被替换为内建类型或class类型：

  • 6个算术运算：

    plus<type>，minus<type>，negate<type>，multiplies<type>，divides<type>，modules<type>

  • 6个关系：

    less<type>，less_ equal<type>，greater<type>，greater_ equal<type>，equal_to<type>，not_equal_to<type>

  • 3个逻辑运算：

    分别对应于&&，||，!运算符的：logical_and<type>，logical_or<type>，logical_not<type>

想要使用事先定义的Function object，首先得包含相关头文件：#include <functional>

在上面的介绍中，一些概念说得其实比较模糊，感觉还是没理解到为什么叫函数对象(function objects)，在[这里](C++函数对象详解（附带实例） (biancheng.net))找到了一个更详细的解释：

如果一个类将()运算符重载为成员函数，这个类就称为函数对象类，这个类的对象就是函数对象。函数对象是一个对象，但是使用的形式看起来像函数调用，实际上也执行了函数调用，因而得名。

自己的理解对函数对象的理解：我们可以用函数对象来自定义一些函数操作，逻辑运算等等，之后，把它传给泛型算法，以让泛型算法去按照我们所想的方式去执行。同时，这么做还有一个效率优点——编译器可以内联执行函数对象的调用。

这个解法的问题在于，大部分情形下，目的端容器的容量显然太大了。另一种解法是先定义一个空容器，而后每当有元素被安插进来时，才加以扩展。不幸的是，filter()目前的实现法是将元素赋值至某个已存在的容器位置上，如果我们重新以安插方式实现filter(), 那对我们目前已有的应用程序会产生什么影响呢?此外，我们又应该提供什么样的安插操作呢?

所有“会对元素进行复制行为”的泛型算法,例如copy()，copy_backwards()，remove_copy()，replace_ _copy()，unique_copy()，等等，都和filter()的实现极为相似。每个算法都接受一个iterator，标示出复制的起始位置。每当复制一个元索时，其值会被赋值(assigned)，iterator则会递增至下个位置。我们必须保证在每一次复制操作中，目的端容器都具有够大的容量以存储这些被赋值进来的元素。既然有了这些算法，我们实在不需要重新写一个。

自己理解的话，这些adapters其实就像是一个个的包装器，当程序编译到这里的时候，会提醒编译器应该用对应的插入操作，而非赋值操作。
